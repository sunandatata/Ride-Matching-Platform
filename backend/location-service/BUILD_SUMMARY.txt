KEY ARCHITECTURAL FEATURES
================================================================================

HIGH-THROUGHPUT DESIGN
  - Batching: 500 updates or 100ms window
  - Pipelined Redis: All commands in single round-trip
  - Async DB: 5-minute batches don't block requests
  - Queue-based: BlockingQueue with circuit breaker
  - Non-blocking events: Kafka publish doesn't fail requests

LOW-LATENCY READS
  - Redis Geo Index: Sub-millisecond spatial queries
  - In-memory batching: < 10ms enqueue latency
  - Cached status: Driver online/offline in Redis (5 min TTL)

CLEAN ARCHITECTURE
  - Thin controller: HTTP handling only
  - Service layer: Business logic orchestration
  - Repository abstraction: JPA for data access
  - DTO validation: Spring validation framework
  - Global exception handler: Centralized error responses

SOLID PRINCIPLES
  - Single Responsibility: Each class has one reason to change
  - Open/Closed: Extensible without modification
  - Liskov Substitution: Proper inheritance
  - Interface Segregation: Focused repositories
  - Dependency Inversion: Constructor injection

API ENDPOINTS
================================================================================

  PUT /api/v1/locations/drivers/location
    Update driver location -> 202 Accepted

  GET /api/v1/locations/nearby?lat=X&lng=Y&radiusKm=R
    Find nearby drivers -> 200 OK with drivers list

  GET /api/v1/locations/drivers/{driverId}/location
    Get current location -> 200 OK or 404 Not Found

  GET /api/v1/locations/drivers/{driverId}/location-history
    Get 1-hour history -> 200 OK with locations list

  GET /api/v1/locations/stats
    Get batch statistics -> 200 OK with metrics
